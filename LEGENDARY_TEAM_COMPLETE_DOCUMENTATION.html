üß† Core Methodologies & Principles

The Legendary Team is built on 10 foundational methodologies that work together to create a revolutionary autonomous engineering system.

1. PRD-First Development (OpenSpec)

Principle: Document before coding. The OpenSpec is the single source of truth.

Implementation:

  • All requirements documented in OpenSpec/ directory
  • @OpenSpecPolice agent enforces "no code without approved specs"
  • Drift protection stops if >15% code missing from specs
  • OpenSpec versioning and rollback capabilities

Why It Matters: Prevents scope creep, enables parallelization (teams know what to build), provides rollback capability.

Golden Rule: NO code implementation without approved OpenSpec. @OpenSpecPolice bans chat TODO lists forever.

2. Modular Rules Architecture

Principle: Split by concern, load context on-demand. No giant monolithic files.

Implementation:

  • .claude/agents/ - Specialized AI teammates (11 agents)
  • .claude/skills/ - Reusable abilities (git, testing, etc.)
  • .claude/commands/ - Slash commands (/review-queue, /team-status)
  • .claude/hooks/ - Automated quality gates
  • Each component is independent and composable

Why It Matters: Reduces context usage, enables team specialization, makes system maintainable at scale.

3. Command-ify Everything

Principle: Make repetitive tasks into reusable slash commands.

Implementation:

  • /bootstrap - Full system startup
  • /review-queue - Display tasks waiting for review
  • /team-status - Monitor active parallel teams
  • /approve-task [id] - Approve queued work
  • /reject-task [id] - Reject and learn
  • /emergency-stop - Kill everything

Why It Matters: Reduces friction, makes workflows reproducible, speeds up common operations.

4. The Context Reset ("Clear, Don't Compact")

Principle: When context fills up, clear it and restore state from ledgers. Never compact (degrades signal).

Implementation:

  • Session ledgers in thoughts/ledgers/ track current work
  • Handoffs in thoughts/shared/handoffs/ transfer knowledge between sessions
  • PreCompact hook blocks compaction, enforces "create handoff first"
  • SessionStart hook auto-loads ledgers and handoffs
  • Complete state restoration after context clear

Why It Matters: Prevents context degradation, enables infinite session length, maintains quality across context resets.

Rule: PreCompact hook will BLOCK compaction until you create a handoff. Use /clear instead.

5. System Evolution Mindset

Principle: Every bug, every failure, every success teaches the system. Continuous learning and adaptation.

Implementation:

  • Artifact index (SQLite) stores all decisions, outcomes, and learnings
  • Confidence model adapts based on task success/failure rates
  • Failed auto-proceed tasks reduce confidence for similar future tasks
  • Successful patterns increase confidence scores
  • "What Didn't Work" sections in handoffs capture failures
  • Searchable history via FTS5 full-text search

Why It Matters: System gets smarter over time, prevents repeated mistakes, institutional knowledge persists.

6. Parallel Autonomous Operation

Principle: High-confidence work proceeds immediately in parallel. Uncertain work queues for async human review. Critical work blocks.

Implementation:

  • 3-Tier Routing:
    • Tier 1 (‚â•70% confidence): Auto-proceed, spawn parallel teams
    • Tier 2 (40-69% confidence): Queue for review, continue other work
    • Tier 3 (<40% confidence): Block for human decision
  • @ConfidenceAgent scores each task (0-100) using 8 weighted factors
  • @chief spawns 3-15 teams simultaneously for independent tasks
  • Review queue managed via thoughts/shared/review-queue.json
  • Teams auto-merge when tests pass

Why It Matters: 3-5x faster delivery, 70% less human overhead, non-blocking workflow.

Example: 10 sub-tasks ‚Üí 5 auto-proceed in parallel (25 min), 4 queue for review, 1 blocks. Result: 90 min total vs 300+ min sequential.

7. Continuity System

Principle: Never lose context. Track session state, transfer knowledge across sessions, remember everything.

Implementation:

  • Ledgers (thoughts/ledgers/): Current session work tracking
    • Goal, constraints, completed work, decisions, next steps
    • Auto-loaded at session start
    • Updated continuously during work
  • Handoffs (thoughts/shared/handoffs/): Cross-session knowledge transfer
    • Changes made, what worked/didn't work, key decisions
    • Permanent record indexed in SQLite
    • Created before context clear or session end
  • Plans (thoughts/shared/plans/): Pre-implementation design
    • Problem, solution, alternatives, risks
    • Referenced by queued tasks

Why It Matters: Infinite session length, seamless handoffs between sessions, institutional memory.

8. Hooks System (Automated Quality Gates)

Principle: Automate quality enforcement. Prevent issues before they happen.

Implementation:

  • SessionStart.js: Auto-load ledgers and handoffs, restore state
  • PreToolUse.js: TypeScript validation, budget checks before tool execution
  • PostToolUse.js: State tracking, artifact indexing, file change monitoring
  • PreCompact.js: Block compaction, enforce "Clear, Don't Compact" philosophy
  • SessionEnd.js: Cleanup, prompt for handoff creation
  • All hooks continue on errors (graceful degradation)

Why It Matters: Quality gates are automatic, prevents mistakes, enforces best practices without human intervention.

9. Artifact Indexing (Searchable History)

Principle: Every decision, every handoff, every plan is searchable. Institutional knowledge is queryable.

Implementation:

  • SQLite database at .claude/cache/artifact-index/artifacts.db
  • FTS5 full-text search on title, summary, content, tags, keywords
  • Indexes: handoffs, plans, learnings, ledgers
  • Tracks outcomes (SUCCEEDED, PARTIAL, FAILED)
  • Auto-indexed by PostToolUse hook
  • Query via SQL or search commands

Why It Matters: Find similar past solutions instantly, learn from history, prevent repeated work.

10. Confidence-Based Routing

Principle: Let data decide routing. Algorithmic confidence scoring determines auto-proceed vs queue vs block.

Implementation:

  • Scoring Algorithm (8 factors):
    • +40: Similar task succeeded before
    • +30: Clear OpenSpec requirements
    • +20: Existing patterns available
    • +10: Low risk (CRUD, UI, tests)
    • -20: Security implications
    • -20: New architectural pattern
    • -30: Conflicting requirements
    • -40: Destructive operations
  • Logged to .claude/cache/confidence-decisions.json
  • Learning loop: outcomes update future confidence
  • False positives degrade confidence by 10%

Why It Matters: Objective routing decisions, self-improving system, transparency in decision-making.

# Check hook configuration

# Edit config.json and set "enabled": true

# Reinitialize database
